Kick off the development pipeline using a vertical-slice approach: each slice is a thin cut through the full stack (DB + backend + frontend), validated in a real browser before the next slice starts. Bugs are caught after a small isolated diff, not after the entire codebase is assembled.

Usage: /build <project-name> [feature-name]

Arguments: $ARGUMENTS

---

## Orchestration Rules

- Agents NEVER talk to each other — all communication through `workspace/{project}/`
- **Within a slice:** backend + frontend + db run in PARALLEL
- **Between slices:** sequential with mandatory gates — do not start the next slice until the current slice's gate passes
- The orchestrator runs ALL validation gates and compile checks directly using Bash — never delegates this
- **Stateless orchestrator:** never open agent-written source files. Verify correctness via mechanical gates (bash pass/fail) only. Reading agent code adds context with no mechanical value — the compile gate already verified it.
- **Checkpoint discipline:** at the start of each slice, read ONLY `workspace/{project}/checkpoints/slice-{N-1}.md`. Do not re-read technical-spec.md, api-spec.yaml, or handoff files — the checkpoint captures all facts needed. Checkpoints also enable resuming a failed build in a fresh context window without restarting from Slice 0.

---

## The Five Engineering Practices

Every slice follows this sequence. No exceptions.

```
1. SLICE CONTRACT  — orchestrator generates a focused 1-page context file per agent
2. TEST-FIRST      — QA writes pytest tests before backend writes any code
3. IMPLEMENT       — backend + frontend + db in parallel, each reading their contract
4. COMPILE GATE    — ruff + py_compile + tsc --noEmit (no Docker needed, catches errors fast)
5. BROWSER GATE    — Playwright confirms the feature works in a real browser
6. COMMIT          — git commit after each passing gate (rollback points + clean diffs)
```

### Practice 1: Slice Contracts

The core context management improvement. Agents currently read 5 large documents and guess what applies to them. A slice contract is generated by the orchestrator before each agent is invoked — it contains ONLY the endpoints, schemas, and pages relevant to that specific slice.

An agent implementing auth reads a 1-page auth contract. It does not read a 200-endpoint API spec and infer which parts are in scope. This is the difference between focused work and context pressure.

Generate contracts by parsing api-spec.yaml with Python (orchestrator runs this directly):

```bash
python3 - <<'PYEOF'
import yaml, os, sys

spec = yaml.safe_load(open('workspace/{project}/api-spec.yaml'))
os.makedirs('workspace/{project}/slices', exist_ok=True)

# Customise SLICE_PREFIXES and SLICE_SCHEMAS per slice
SLICE_PREFIXES = ['/api/v1/auth/', '/api/v1/dev/']
SLICE_SCHEMAS  = ['User', 'LoginRequest', 'RegisterRequest']

slice_paths = {k: v for k, v in spec.get('paths', {}).items()
               if any(k.startswith(p) for p in SLICE_PREFIXES)}
slice_schemas = {k: v for k, v in spec.get('components', {}).get('schemas', {}).items()
                 if k in SLICE_SCHEMAS}

with open('workspace/{project}/slices/slice-1-backend.md', 'w') as f:
    f.write(f"# Slice 1: Auth — Backend Contract\n\n")
    f.write(f"## Endpoints to implement\n")
    f.write(yaml.dump({'paths': slice_paths}, default_flow_style=False))
    f.write(f"\n## Schemas\n")
    f.write(yaml.dump({'schemas': slice_schemas}, default_flow_style=False))
    f.write(f"\n## Already built (DO NOT re-implement)\n")
    f.write(f"- FastAPI app factory, health endpoint, config, database session (Slice 0)\n")
    f.write(f"\n## Do NOT implement\n")
    f.write(f"- Any non-auth endpoints\n")
    f.write(f"\n## Pre-written tests — your implementation must make these pass\n")
    f.write(f"workspace/{{project}}/slices/slice-1-tests/test_auth.py\n")
PYEOF
```

Adapt `SLICE_PREFIXES` and `SLICE_SCHEMAS` for each slice. The frontend contract lists pages, API shapes, and what's already built.

### Practice 2: Test-First

For every backend slice (except Slice 0), invoke `qa-engineer` **before** `backend-engineer`. Give it the slice's backend contract. It writes pytest tests covering every endpoint. Then backend-engineer reads both the contract and the tests, and implements until they pass.

This gives the backend-engineer a concrete, runnable definition of "done" that doesn't require subjective interpretation of a spec. It also catches API shape disagreements before a line of implementation is written.

### Practice 3: Compile Gate (runs without Playwright)

After agents complete, before any browser test:

```bash
# Backend — in Docker (deps installed there)
docker compose exec backend bash -c "
  ruff check app/ --quiet         && echo '[PASS] ruff'       || { echo '[FAIL] ruff';     exit 1; }
  python -m compileall app/ -q   && echo '[PASS] py_compile' || { echo '[FAIL] syntax';   exit 1; }
  pytest slices/slice-N-tests/ -x -q 2>&1 | tail -20 \
                                  && echo '[PASS] tests'       || { echo '[FAIL] tests';    exit 1; }
"

# Frontend — in Docker (node_modules installed there)
docker compose exec frontend bash -c "
  npx tsc --noEmit --pretty 2>&1 | tail -20 \
  && echo '[PASS] TypeScript' || { echo '[FAIL] type errors above'; exit 1; }
"
```

A type error here takes 30 seconds to fix. The same error found during Playwright takes 15 minutes. Compile gate failures block the browser gate — fix them first.

### Practice 4: OpenAPI → TypeScript Types

After Slice 1's browser gate passes, generate TypeScript types from the live backend. Regenerate after every subsequent backend slice.

```bash
curl -s http://localhost:8000/openapi.json \
  | npx openapi-typescript /dev/stdin \
  -o workspace/{project}/src/frontend/src/types/api-generated.ts

echo "[PASS] api-generated.ts updated from live OpenAPI"
```

All frontend agents from Slice 2 onwards **must** import types from `api-generated.ts`. They must not hand-write types for API responses. This makes type mismatches TypeScript compile errors — caught at the compile gate, not in the browser.

### Practice 5: Git Commit After Each Passing Gate

```bash
cd workspace/{project}/src
git add -A
git commit -m "slice-{N}-{name}: gate passed — {N} tests, {endpoints} endpoints"
```

If Slice 3 breaks something from Slice 1: `git diff slice-1-auth..slice-3-feature` shows exactly what changed. `git checkout slice-1-auth` returns to a known-good state. Without commits between slices, debugging a regression means diffing the entire build.

## Checkpoint Protocol

After every gate passes, the orchestrator writes a checkpoint. This is the orchestrator's external memory — it replaces reading handoffs, specs, or prior agent outputs in subsequent slices.

**Format** — write to `workspace/{project}/checkpoints/slice-{N}.md`:

```markdown
# Slice {N}: {name} — Checkpoint

**Completed:** {ISO datetime}
**Git SHA:** {output of: git -C workspace/{project}/src rev-parse --short HEAD}
**Prior checkpoint:** slice-{N-1}.md

## Seed credentials
- {email} / {password} (role: {role})
- {email} / {password} (role: {role})

## Schema facts (authoritative — do not re-derive from specs)
- Enum: {field} ∈ [{values confirmed in this slice}]
- Response envelope: {endpoint} → {exact shape}
- Critical column names: {any non-obvious names that caused bugs}

## Bugs fixed this slice
- {short description} → {fix applied}

## Agent files updated
- {agent-file.md}: {rule added}

## Gate results
- Compile: PASS ({N} tests, ruff clean, tsc clean)
- Browser: PASS ({what was verified})
```

**Reading checkpoints:** At the start of each slice, run:
```bash
ls workspace/{project}/checkpoints/
cat workspace/{project}/checkpoints/slice-{N-1}.md
```

This is the ONLY file the orchestrator reads at slice start. All facts from prior slices — enum values, seed credentials, bugs fixed, agent updates — are here. Do not open any other file for prior-state context.

**Resuming a failed build:** If the orchestrator's context window fills mid-build, start a new `/build {project}` session. At STEP 1, check `ls workspace/{project}/checkpoints/` — the highest-numbered checkpoint is the resume point. Read it, skip to the next slice.
